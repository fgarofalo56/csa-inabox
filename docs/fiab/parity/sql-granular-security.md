# sql-granular-security — parity with column-/row-level security in Synapse SQL & Databricks Unity Catalog

Source UI:
- Azure Synapse / SQL — column-level security (GRANT on column list):
  https://learn.microsoft.com/azure/synapse-analytics/sql-data-warehouse/column-level-security
- Azure SQL / Synapse Dedicated — Row-Level Security (security policy + predicate):
  https://learn.microsoft.com/sql/t-sql/statements/create-security-policy-transact-sql
- Databricks SQL editor — Unity Catalog **column masks**:
  https://learn.microsoft.com/azure/databricks/sql/language-manual/sql-ref-syntax-ddl-column-mask
- Databricks SQL editor — Unity Catalog **row filters**:
  https://learn.microsoft.com/azure/databricks/data-governance/unity-catalog/filters-and-masks/manually-apply

This surface extends the Permissions experience beyond object RBAC. Two backends,
both Azure-native (no Microsoft Fabric / Power BI workspace required):

- **Synapse / Warehouse** — `SqlSecurityPanel` (`lib/panes/sql-security-panel.tsx`)
  opens from `WarehouseEditor` (Home ▸ Security ▸ "Column & Row security",
  `lib/editors/phase3-editors.tsx`) and from the Synapse Dedicated / Serverless
  SQL pool editors. Generated by `lib/sql/tsql-builders.ts`; executed over TDS +
  Microsoft Entra token by `app/api/items/[type]/[id]/sql-security/route.ts`.
- **Databricks Unity Catalog** — `UcSecurityPanel` (`lib/panes/uc-security-panel.tsx`)
  opens from `DatabricksSqlWarehouseEditor` (Home ▸ Unity Catalog ▸ "Column & row
  security", `lib/editors/databricks-editors.tsx`). Generated by
  `lib/sql/uc-security-builders.ts`; executed over the Databricks SQL Statement
  Execution API by `app/api/items/[type]/[id]/security/route.ts`.

## Feature inventory (the real UIs)

### Synapse / Azure SQL granular security
| # | Capability | Notes |
|---|------------|-------|
| 1 | Object-level GRANT / DENY | `GRANT <perms> ON OBJECT::[s].[o] TO [principal]`. |
| 2 | Column-level GRANT / DENY | `GRANT SELECT ON [s].[t](col,…) TO [principal]` → denied columns raise Msg 230. |
| 3 | Row-Level Security | Inline TVF predicate + `CREATE SECURITY POLICY … ADD FILTER PREDICATE … WITH (STATE=ON)`. |
| 4 | Dynamic Data Masking | `ALTER COLUMN … ADD MASKED WITH (FUNCTION='…')`. |
| 5 | Verify as principal | `EXECUTE AS USER='upn'; SELECT …; REVERT;` to prove the effect. |
| 6 | Current state | `sys.database_permissions`, `sys.masked_columns`, `sys.security_policies`. |

### Databricks Unity Catalog granular security
| # | Capability | Notes |
|---|------------|-------|
| A | Column mask | `CREATE FUNCTION` (CASE over `IS_ACCOUNT_GROUP_MEMBER`) + `ALTER TABLE … ALTER COLUMN … SET MASK`. |
| B | Clear column mask | `ALTER TABLE … ALTER COLUMN … DROP MASK`. |
| C | Row filter | `CREATE FUNCTION` (`CURRENT_USER() = col OR IS_ACCOUNT_GROUP_MEMBER`) + `ALTER TABLE … SET ROW FILTER … ON (col)`. |
| D | Drop row filter | `ALTER TABLE … DROP ROW FILTER`. |
| E | Verify binding | Read `information_schema.column_masks` / `.row_filters`; spot-check SELECT (admin view). |
| F | Current state | `information_schema.column_masks`, `information_schema.row_filters`. |
| G | Schema / table / column pickers | `information_schema.schemata` / `.tables` / `.columns`. |

## Backend per control

| Control | Backend call |
|---------|--------------|
| Synapse Column GRANT / RLS / DDM execute | TDS + Entra token (`synapse-sql-client` / `azure-sql-client`) via `sql-security/route.ts`. |
| Synapse Verify-as-principal | `EXECUTE AS USER … SELECT … REVERT` over the same TDS connection. |
| UC mask / filter apply | `CREATE OR REPLACE FUNCTION` then `ALTER TABLE …` via `executeStatement` (Databricks SQL Statement Execution API, MI bearer token). |
| UC drop mask / filter | single `ALTER TABLE …` statement via `executeStatement`. |
| UC verify | `SELECT * … LIMIT n` (admin view) + `information_schema.column_masks` / `.row_filters` reads. |
| UC pickers | `information_schema.schemata` / `.tables` / `.columns` reads. |

## Principal model

- **Synapse**: principals are contained database users mapped from Entra (`CREATE
  USER [upn] FROM EXTERNAL PROVIDER`). The RLS predicate compares the filter column
  to `USER_NAME()`; column GRANT targets the user/role by name.
- **Databricks UC**: principals are Microsoft Entra UPNs (SCIM-synced). Inside a UC
  UDF, `CURRENT_USER()` returns the invoking user's UPN and
  `IS_ACCOUNT_GROUP_MEMBER('<group>')` checks account-group membership.

## Verification

Acceptance criteria are met as follows:

- **Column-level grant restricts a real test query to the allowed columns
  (Synapse)** — Column tab → Execute `GRANT SELECT ON [s].[t](cols) TO [upn]`, then
  "Verify as principal" runs `EXECUTE AS USER='upn'; SELECT TOP n * FROM [s].[t];
  REVERT;`. The result grid shows only the granted columns (denied columns raise
  Msg 230 in the receipt).
- **RLS hides rows for the constrained principal (Synapse)** — RLS tab → Execute the
  security policy, then "Verify rows visible" runs a `COUNT(*)`/`SELECT` under
  `EXECUTE AS`. Before-policy: full count; after-policy: only the principal's rows.
- **Receipt shows before/after for the constrained principal (Databricks UC)** —
  the `verify` action returns the admin (UAMI) view rows, the
  `information_schema.column_masks` / `.row_filters` rows proving the binding is
  live, and a note explaining that the mask/filter UDF evaluates `CURRENT_USER()` /
  `IS_ACCOUNT_GROUP_MEMBER()` at query time, so the constrained principal sees the
  masked values / filtered rows when they run the same SELECT.
