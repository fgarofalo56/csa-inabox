## Problem

The cross-item Copilot (`/api/copilot/orchestrate` + the right-rail
`CopilotPane`) used a single fixed `SYSTEM_PROMPT` and exposed every tool to
every surface. The pane title was hard-coded "Copilot". The model had no
knowledge of the query/cell the user was editing.

## Design

A **persona registry** (`lib/azure/copilot-personas.ts`) maps a
`contextSlug` (pane) â†’ `{ title, greeting, systemPrompt(payload), toolCatalog,
suggestedPrompts }`. The orchestrator selects the persona **server-side** and
composes the system prompt from an injected `contextPayload` (active query,
schema, workspace id, item id). Tools are filtered to the persona's catalog
(empty catalog = all tools = the backward-compatible `default` persona).

Slugs: `default`, `notebook`, `warehouse`, `lakehouse`, `data-pipeline`,
`kql-database`, `data-agent`.

## No-Fabric / no-vaporware notes

- Azure-native only: the backend is Azure OpenAI; no `api.fabric.microsoft.com`
  / `api.powerbi.com` is touched. Works with `LOOM_DEFAULT_FABRIC_WORKSPACE`
  unset.
- No new Azure resource / env var / Cosmos container / role assignment â€” this is
  a code-layer routing change, so **no bicep change is required**.
- No hard-coded reply: every persona's `systemPrompt` is a template function; the
  answer is generated by AOAI grounded in the injected payload.
- `suggestedPrompts` + `toolCatalog` are fixed in-code allowlists (per
  loom-no-freeform-config) â€” not admin-configurable, not stored.
