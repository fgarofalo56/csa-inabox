# Visual tutorials — per item, app & feature

Step-by-step screenshot walkthroughs for **every CSA Loom item type, curated
app, and console feature page**, generated by the tutorial-capture UAT
(`apps/fiab-console/e2e/tutorial-capture.uat.ts`) driving the **live** Loom
console. Each run doubles as a functional UAT: it opens the surface, dismisses
the "Learn about this item" drawer, and captures one screenshot per tab/step.

This index is **regenerated automatically** by
`scripts/csa-loom/publish-tutorials.mjs` each time reviewed captures are
published — it will list every published tutorial grouped by dimension
(`item-*`, `app-*`, `feature-*`).

## How the set is produced

Live-console screenshots can contain Azure resource names, so they go through a
mandatory privacy gate — they are **staged** for review, then published by an
operator (never auto-committed):

1. **Capture** (CI: `csa-loom-tutorial-capture` workflow, or locally with a
   minted session) stages images into
   `temp/azure-screenshots/redacted/loom-tutorials/`.
2. **Review** the staged `MANIFEST.md` and images for sensitive data.
3. **Publish**: `node scripts/csa-loom/publish-tutorials.mjs` copies approved
   slugs here and regenerates this index.
4. **Audit**: `node scripts/csa-loom/check-tutorial-coverage.mjs` reports any
   surface still missing a tutorial (run with `--strict` in CI to block drift).

See the full [capture + maintenance process](../CAPTURE-PROCESS.md).

## Coverage

The expected set is parsed from source (no hand-maintained list):

- **Items** — every `reg(...)` entry in `apps/fiab-console/lib/editors/registry.ts` (103).
- **Features** — every page in `NAV_PAGES` (`apps/fiab-console/e2e/_lib/uat.ts`) (17).
- **Apps** — every curated app from `GET /api/apps-catalog` (resolved at capture time).
