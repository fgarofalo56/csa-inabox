## 1. Fabric capability inventory (grounded in MS Learn)

### 1.1 Lakehouse
- **Item model.** A lakehouse is a OneLake folder structure with two top-level zones: `Tables/` (managed Delta
  Parquet, auto-registered to the metastore) and `Files/` (raw/unstructured). Each lakehouse auto-provisions a
  **SQL analytics endpoint** (read-only TDS over the Delta tables) and (pre-Sep-2025) a default semantic model
  (now decoupled). Schema-enabled lakehouses add a schema layer under `Tables/`.
  - https://learn.microsoft.com/fabric/data-engineering/lakehouse-overview
  - https://learn.microsoft.com/fabric/fundamentals/store-data
- **Automatic table discovery.** Files landed in `Tables/` are validated (Delta only), metadata-extracted, and
  registered so they're immediately queryable in Spark SQL and T-SQL.
- **Load to Tables.** UI "Load to Tables" converts CSV/Parquet files into managed Delta tables.
  - https://learn.microsoft.com/fabric/data-engineering/load-to-tables
- **OneLake shortcuts.** Named zero-copy pointers to external storage (ADLS, S3, GCS, Dataverse, other
  lakehouses). Delta shortcuts in `Tables/` surface in the SQL endpoint.
  - https://learn.microsoft.com/fabric/onelake/onelake-shortcuts
- **OneLake security / table maintenance / time travel.** Object/table/column/row security; Delta time-travel
  (VERSION AS OF) and RESTORE.

### 1.2 SQL analytics endpoint
- Read-only T-SQL/TDS engine (same compute as Fabric Warehouse) over the lakehouse Delta tables. Auto metadata
  sync (background process, <1 min lag, halts after 15 min idle). Supports user views, SPs, functions, but the
  autogenerated tables are read-only. Only Delta tables in `Tables/` are exposed.
  - https://learn.microsoft.com/fabric/data-engineering/lakehouse-sql-analytics-endpoint
  - https://learn.microsoft.com/fabric/data-engineering/sql-analytics-endpoint-metadata-sync
  - https://learn.microsoft.com/fabric/data-engineering/sql-analytics-endpoint-performance
- **Visual query** (diagram drag-drop, Power Query online) + SQL query editor in the portal.

### 1.3 Spark notebooks
- **Multi-language** primary kernel: PySpark (Python), Spark (Scala), Spark SQL, SparkR; per-cell language
  **magics** `%%pyspark %%spark %%sql %%sparkr %%html`. IDE IntelliSense, snippets, drag-drop, markdown folding.
  - https://learn.microsoft.com/fabric/data-engineering/author-execute-notebook
- **notebookutils / mssparkutils.** Built-in package: `fs` (ADLS/Blob/mount), `notebook.run` / `runMultiple`
  (parallel + topological DAG of child notebooks on isolated REPLs), `credentials.getSecret/getToken`, `env`,
  `nbResPath` (notebook resources).
  - https://learn.microsoft.com/fabric/data-engineering/notebook-utilities
  - https://learn.microsoft.com/fabric/data-engineering/microsoft-spark-utilities
- **Notebook resources.** Per-notebook Unix-like file system (`builtin/`) for small files usable at runtime;
  referenced via `mssparkutils.nbResPath`.
- **Built-in scheduling.** Notebooks have native scheduled-run support (Fabric advantage over Synapse).
- **High concurrency mode.** Multiple notebooks share one Spark session (REPL-core isolation, FAIR scheduling),
  within single-user boundary + same default lakehouse + same compute + same libraries. Up to 5 notebooks/session
  (tunable to 50 via `spark.highConcurrency.max`). ~36x faster subsequent session start on custom pools.
  - https://learn.microsoft.com/fabric/data-engineering/high-concurrency-overview
  - https://learn.microsoft.com/fabric/data-engineering/configure-high-concurrency-session-notebooks
- **Copilot in notebook** (NL→code, explain), **Data Wrangler** launch (see 1.7).

### 1.4 Spark job definition (SJD)
- Code item submitting batch/streaming jobs: main definition file (.py/.jar/.R), reference libs, command-line
  args, referenced + **default lakehouse context**, Spark properties. Run manual/scheduled; runs grid with
  near-real-time status; cancel active run; **snapshot** (save/open/restore the submitted state); driver logs.
  - https://learn.microsoft.com/fabric/data-engineering/spark-job-definition
  - https://learn.microsoft.com/fabric/data-engineering/run-spark-job-definition

### 1.5 Environments
- Three components: **Spark compute** (runtime version + session-level driver/executor cores+memory + pool),
  **Libraries** (built-in + public pip/conda + custom .whl/.jar; Quick vs Full publish mode), **Resources**
  (real-time shared files, not publish-gated). Set as workspace default. Save+Publish required.
  - https://learn.microsoft.com/fabric/data-engineering/create-and-use-environment
  - https://learn.microsoft.com/fabric/data-engineering/environment-manage-compute
  - https://learn.microsoft.com/fabric/data-engineering/environment-manage-library

### 1.6 Spark compute / pools
- **Starter pools** (prewarmed, ~5s start, autoscale on by default). **Custom pools** (node family/size
  Small→XX-Large, autoscale min/max ≤200, dynamic executor allocation, 2-min autopause). **Custom live pools**
  (prehydrated, ~5s start with Full-mode libs). Workspace admin Spark settings: default pool/env, runtime, high
  concurrency, **job admission** (optimistic vs reserve-max-cores), **session timeout** (default 20 min),
  autologging.
  - https://learn.microsoft.com/fabric/data-engineering/create-custom-spark-pools
  - https://learn.microsoft.com/fabric/data-engineering/spark-compute
  - https://learn.microsoft.com/fabric/data-engineering/workspace-admin-settings

### 1.7 Data Wrangler
- Notebook-based interactive EDA: launch on a pandas **or Spark** DataFrame, browse columns + summary stats +
  distributions, apply common cleaning operations (filter, drop, rename, one-hot encode, fillna, group-by, sort,
  type cast) via a panel with **real-time preview**, and **export generated code** (pandas or translated PySpark)
  back into a notebook cell as a function. AI/Copilot codegen in newer builds.
  - https://learn.microsoft.com/fabric/data-science/data-wrangler
  - https://learn.microsoft.com/fabric/data-science/data-wrangler-spark
  - https://learn.microsoft.com/fabric/data-science/data-wrangler-ai

### 1.8 Livy API
- Per-lakehouse REST endpoint to submit Spark **session** jobs (persistent, interactive, shared state),
  **batch** jobs (one-shot), and **high-concurrency** sessions (independent REPL contexts) from remote clients
  (VS Code, curl) with Entra user/SPN tokens. Connection string surfaced in Lakehouse Settings → Livy endpoint.
  `Code.*` scopes gate downstream access (Fabric, Storage, KeyVault, ADX, SQL).
  - https://learn.microsoft.com/fabric/data-engineering/api-livy-overview
  - https://learn.microsoft.com/fabric/data-engineering/get-started-api-livy

### 1.9 Native Execution Engine (NEE)
- Vectorized C++ engine (Meta **Velox** + Intel **Apache Gluten**) offloading supported operators from JVM Spark;
  columnar SIMD, native Parquet/Delta, preserves AQE/pruning/pushdown; up to ~4-6x TPC-DS speedup, no code
  changes. Enabled per-environment (Spark Compute → **Acceleration** toggle) or per-job (`spark.native.enabled`).
  Falls back to JVM for unsupported ops (structured streaming, JSON/XML, ANSI mode).
  - https://learn.microsoft.com/fabric/data-engineering/native-execution-engine-overview
  - https://learn.microsoft.com/fabric/data-engineering/runtime

### 1.10 Table maintenance
- Delta `OPTIMIZE` (bin-compaction) + optional **V-Order** (write-time Parquet layout, ~15% slower writes / up
  to 50% more compression / faster Power BI + SQL reads) + **Z-Order**, `VACUUM` (default 7-day retention,
  rejects <7d), PURGE (deletion-vector merge). Ad-hoc (Lakehouse Explorer Maintenance dialog), REST (Table
  Maintenance API, async poll), or scheduled (**Lakehouse Maintenance activity** in pipelines). Auto-compaction
  table properties (`delta.autoOptimize.*`). Health-driven via `sp_get_table_health_metrics`.
  - https://learn.microsoft.com/fabric/data-engineering/lakehouse-table-maintenance
  - https://learn.microsoft.com/fabric/data-engineering/delta-optimization-and-v-order
  - https://learn.microsoft.com/fabric/data-engineering/lakehouse-api

### 1.11 Materialized Lake Views (MLV)
- Declarative `CREATE MATERIALIZED LAKE VIEW` with auto-refresh + DAG lineage centralized in one lakehouse.
  - https://learn.microsoft.com/fabric/data-engineering/spark-best-practices-development-monitoring

### 1.12 API for GraphQL
- Visual builder: create GraphQL item → connect data source (Lakehouse SQL endpoint / Warehouse / SQL DB /
  Mirrored DB) from OneLake catalog → pick tables/views/SPs → optional relationships → **Fabric auto-generates
  schema + resolvers** + endpoint. SSO or saved-credential connectivity; Entra-only client auth; multi-source
  fan-out queries; stored-procedure mutations.
  - https://learn.microsoft.com/fabric/data-engineering/api-graphql-overview
  - https://learn.microsoft.com/fabric/data-engineering/get-started-api-graphql
  - https://learn.microsoft.com/fabric/data-engineering/multiple-data-sources-graphql

### 1.13 Spark monitoring
- Monitoring Hub (all Spark activities), Spark UI / history server, **Spark Advisor** (real-time alerts incl.
  NEE fallback), driver/executor logs, run snapshots.

### GAP 9 — Data Wrangler (interactive transform + code generation) — **P1, missing**
**Architecture (words).** A new `DataWranglerPanel` launches from the notebook ribbon (and from the lakehouse
Preview tab) on a selected DataFrame variable or Delta table. The BFF samples N rows (configurable: first/last/
random) into a server-side pandas frame (for a Spark DF, sample → pandas, exactly like Fabric). The panel renders
a column grid + per-column summary stats + mini-distribution. An **Operations rail** (dropdown-driven, no
freeform) offers: filter, drop columns, rename, change type, fill missing, one-hot encode, group-by/aggregate,
sort, dedupe, split column, replace. Each applied op appends to a **recipe** (ordered, reorderable, removable)
and shows a **real-time preview diff** against the sample. On export, the recipe compiles to a function in the
target language — **pandas for a pandas DF, translated PySpark for a Spark DF** — inserted as a new notebook cell
(it never overwrites the source DF, matching Fabric). Sampling + preview run in an ACA "wrangler" worker (pandas)
or via a short Livy `runMultiple` for Spark-side type fidelity.

**UI (Web-5.0).** Split pane: left = data grid (virtualized, sortable, numeric-aware) with column header chips
(type icon + null %); right = Operations rail (Fluent `Accordion` of op groups) + recipe `List` with drag handles;
top toolbar = sample-size dropdown, sampling-method dropdown, language badge (auto from source), Export-to-cell +
Copy-code. Empty state via `EmptyState`. Honest gate (warning MessageBar) if the wrangler ACA app or Spark backend
is unset, naming the env var.

**BFF APIs.** `POST /api/wrangler/sample` `{source, kind, size, method}` → `{columns, rows, stats}`;
`POST /api/wrangler/preview` `{recipe}` → preview rows + affected counts; `POST /api/wrangler/codegen`
`{recipe, lang}` → generated source. All return `{ok, data, error}`.

**Azure services.** ACA "loom-wrangler" (Python: pandas + pyarrow + a pandas→PySpark transpiler for the supported
op set); Synapse Spark/Databricks (Livy) for Spark-side sampling/preview; ADLS for Delta reads.

**Deploy (bicep).** New module `modules/data-plane/wrangler-aca.bicep` (ACA app + UAMI `Storage Blob Data Reader`
+ scale-to-zero). Add `LOOM_WRANGLER_URL` to `apps[]` env in `admin-plane/main.bicep`. Day-one: deployed + enabled.

**Commercial vs Government.** Identical: pandas/pyarrow/ACA all exist in Gov. Spark path uses Synapse/Databricks
Gov (`*.usgovcloudapi.net`). No AOAI dependency for the base wrangler. The **AI/Copilot codegen** add-on uses Gov
AOAI where available, else hides the AI button (capability still works without it).

**Day-one config.** Wrangler ACA deployed and `LOOM_WRANGLER_URL` set; ribbon button live on every notebook +
lakehouse Preview tab.

**Acceptance.** Launch on a real Delta table with `LOOM_DEFAULT_FABRIC_WORKSPACE` unset → apply filter + one-hot
+ fillna → preview updates live → Export to cell inserts runnable PySpark → cell runs on Spark and writes a Delta
table. Read-only grid keeps existing copy-CSV. Remove the "1:1 Data Wrangler" claim from `delta-preview-grid.tsx`
(or repoint it at this panel).

### GAP 10 — API for GraphQL visual auto-schema builder — **P1, stubbed**
**Architecture.** Replace the freeform-SDL textarea path with a **visual builder over Data API Builder (DAB)**
(OSS, Microsoft). Wizard: (1) pick data source from the Loom OneLake-style catalog (lakehouse SQL endpoint /
warehouse / SQL DB / mirrored DB); (2) introspect the source (Synapse Serverless `INFORMATION_SCHEMA`) → present
tables/views/SPs as a **checkable tree**; (3) optionally define relationships between selected entities; (4) Loom
generates the **DAB config (`dab-config.json`) → auto GraphQL + REST schema + resolvers**, deploys it to an ACA
DAB runtime, and returns the endpoint. SDL is generated, not hand-written (honors `loom_no_freeform_config`; the
generated SDL is shown read-only with a "regenerate" action). Multi-source = multiple DAB data-sources / fan-out.

**UI.** 4-step wizard (`Stepper`): Source picker (cards) → Object tree (checkboxes, search/filter) → Relationships
(canvas: drag entity→entity, pick cardinality) → Review (read-only generated SDL + endpoint + test console with
Monaco GraphQL). Permissions tab maps roles→entity actions (dropdowns).

**BFF APIs.** `GET /api/items/graphql-api/[id]/datasources`; `POST .../introspect` `{sourceId}` →
`{tables, views, procs}`; `POST .../generate` `{entities, relationships, perms}` → `{sdl, dabConfig}`;
`POST .../publish` → deploys DAB to ACA, returns endpoint; `POST .../query` (already exists) runs against it.

**Azure services.** ACA "loom-dab" (microsoft/data-api-builder image) + UAMI; Synapse Serverless / SQL for
introspection + resolvers; config stored in Cosmos item state.

**Deploy.** `modules/data-plane/dab-aca.bicep`; `LOOM_DAB_URL` in `apps[]`. Day-one ON.

**Commercial vs Government.** DAB is OSS + container — runs in both clouds; no Fabric/Power BI dependency. Entra
client auth uses Gov authority in Gov.

**Acceptance.** Connect a real Synapse Serverless endpoint, select 2 tables + a view, define one relationship,
generate → publish to ACA → test query returns real rows; multi-source fan-out across two sources returns combined
data. Keep the APIM-fronted path as an opt-in alternative.

### GAP 11 — Native Execution Engine acceleration — **P1, missing**
**Architecture.** Loom's 1:1 for Velox/Gluten NEE is **Databricks Photon** (vectorized C++ engine, managed,
supported). Add an **Acceleration** control to the Spark Environment editor and a per-notebook/per-SJD toggle. On
the Databricks backend, enabling acceleration sets the cluster/job to a Photon-enabled runtime. On the Synapse
backend (no Photon), surface the toggle as **honest, deployed**: enable OSS Gluten+Velox session conf where the
pool supports it, else show a styled note "Acceleration available on the Databricks compute backend" with a
one-click switch — never a dead control.

**UI.** New "Acceleration" tab in `spark-environment-editor.tsx` (Fluent `Switch` + explainer + supported-ops
hint + fallback caveat). Notebook session-config dialog gains an "Enable acceleration (Photon/NEE)" switch that
writes `spark.native.enabled`/Photon into the `%%configure` equivalent.

**BFF APIs.** Extend env publish + notebook session-config payloads with `acceleration: boolean`; map to Photon
runtime (Databricks Clusters API) or `spark.native.enabled` conf (Synapse Livy).

**Azure services.** Databricks (Photon) primary; Synapse Spark (Gluten conf) secondary.

**Deploy.** No new infra; flag plumbed through existing env/session routes. Day-one: acceleration **ON by default**
on the Databricks backend (user can disable).

**Commercial vs Government.** Azure Databricks + Photon are available in both Commercial and Government. Synapse
Spark in both. No managed-service gap.

**Acceptance.** Toggle acceleration on a Databricks-backed notebook → run a TPC-style aggregation → receipt shows
Photon-enabled runtime + measurable speedup; toggling off reverts. On Synapse backend the toggle is functional or
shows the honest switch — never dead.

### GAP 12 — High-concurrency session sharing + notebook orchestration (runMultiple/DAG) — **P1, missing**
**Architecture.** Two parts. (a) **HC session sharing**: a workspace Spark setting + per-notebook "share session"
option so notebooks with the same compute + default lakehouse + user reuse one Livy/Databricks session (Synapse
session reuse; Databricks shared/high-concurrency cluster). (b) **Notebook orchestration**: a `runMultiple` 1:1 —
a canvas where the user composes a **DAG of notebooks** (nodes = notebooks, edges = dependencies, per-node params)
and runs them in parallel/topological order on a shared session, with a live run grid + per-node snapshots. This
is the Azure-native equivalent of `mssparkutils.notebook.runMultiple()`.

**UI.** Reuse the existing canvas-node-kit: a "Notebook Orchestration" canvas (drag notebooks, draw deps, set
params + concurrency), Run/Cancel, live status chips, per-node log drawer. HC toggle lives in the new Workspace
Spark settings page (GAP 14) + as a session-config switch.

**BFF APIs.** `POST /api/notebook/orchestrate` `{nodes, edges, sessionTag, maxConcurrency}` → submits a driver
notebook that fans out child notebooks on REPL cores (Synapse) / parallel jobs (Databricks); `GET .../orchestrate/[runId]`
→ per-node status + snapshots. Session-config gains `shareSession`/`sessionTag`.

**Azure services.** Synapse Spark (REPL multiplexing via a driver notebook) / Databricks (jobs with
`run_job_task` DAG); Livy transport.

**Deploy.** No new infra; routes + canvas. Day-one ON (orchestration available; HC default enabled at workspace
level, disableable).

**Commercial vs Government.** Both backends in both clouds; no gap.

**Acceptance.** Build a 3-notebook DAG (A→B,C) with params, run → B and C run in parallel after A on one shared
session → run grid shows snapshots + exit values; HC toggle measurably cuts subsequent session-start time.

### GAP 13 — Notebook resources file system (nbResPath) — **P1, missing**
**Architecture.** A per-notebook resource folder backed by a dedicated ADLS path
(`loom-notebooks/<notebookId>/resources/`). UI = a file tree in the notebook side panel (upload/new-folder/delete/
download). At session start the BFF mounts/syncs the folder so code can reference it via a Loom-provided
`nbResPath` equivalent (a session env var/path injected into the kernel). Files are real-time (no publish step),
matching Fabric.

**UI.** New "Resources" panel in the notebook editor (Fluent `Tree` + toolbar). `EmptyState` when empty.

**BFF APIs.** `GET/POST/DELETE /api/items/notebook/[id]/resources` (ADLS-backed); session bootstrap injects the
path.

**Azure services.** ADLS Gen2 (existing DLZ storage) + Livy session bootstrap.

**Deploy.** No new infra (reuse DLZ storage); container `loom-notebooks` created via cosmos/storage init. Day-one ON.

**Commercial vs Government.** ADLS in both; no gap.

**Acceptance.** Upload `config.yaml` to a notebook's resources → a cell reads it via the injected path on a real
Spark session with no Fabric bound.

**Azure services.** Synapse pools (ARM), Cosmos.

**Deploy.** No new infra; surfaces existing settings. Day-one: page present, sensible defaults applied

**Commercial vs Government.** Synapse + Cosmos in both; no gap.
