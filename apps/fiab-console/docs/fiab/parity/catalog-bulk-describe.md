# catalog-bulk-describe — parity with Fabric OneLake catalog "AI-generated descriptions"

Source UI: Microsoft Fabric OneLake catalog item detail + Power BI / Fabric
semantic-model "Create descriptions with Copilot" (bulk auto-describe all
tables/measures of a model) —
https://learn.microsoft.com/fabric/governance/onelake-catalog-overview
and Copilot description authoring —
https://learn.microsoft.com/power-bi/create-reports/copilot-introduction

Loom surface: **Catalog → asset detail** (`/catalog/onelake/[id]`) → Overview tab
→ **Cross-source actions** card → **"Generate AI descriptions"** section
(`lib/components/catalog/cross-source-actions.tsx`, `source='onelake'`).
Route: `POST /api/catalog/describe`. Generator: `lib/copilot/bulk-describe.ts`
(shared with the DAX Copilot's per-measure `dax_describe_model` tool — one source
of truth for the AOAI batch plumbing).

> **No Microsoft Fabric / Power BI on the default path.** Descriptions are
> generated by Azure OpenAI (cloud-aware via `resolveAoaiTarget` → `cogScope()`)
> and the model metadata is read from / written to the Cosmos `items` container
> (`item.state.content` for tables+columns+measures, `item.state.model` for
> editor-authored measures). Zero `api.fabric.microsoft.com` / `api.powerbi.com`
> calls — works with `LOOM_DEFAULT_FABRIC_WORKSPACE` UNSET.

## Fabric/Power BI feature inventory

| # | Capability (source UI)                                  | What it does |
|---|---------------------------------------------------------|--------------|
| 1 | Bulk-describe all measures                               | One action proposes descriptions for every measure on the model |
| 2 | Bulk-describe all table columns                          | One action proposes descriptions for every table column |
| 3 | Preview before write                                     | Show proposed descriptions; nothing is persisted until confirmed |
| 4 | Apply / write back to the model                          | Persist the approved descriptions onto the model object metadata |
| 5 | Fill-blank vs overwrite                                  | Only fill empty descriptions, or replace existing ones |
| 6 | Reachable from the catalog detail                        | The bulk action lives on the catalog item detail, not buried in an editor |

## Backend per control

- **Preview / Generate** → `POST /api/catalog/describe` `{ itemId, itemType:'semantic-model', apply, targets, overwrite }`.
  - Reads the content-backed item via `loadContentBackedItem` (tolerates the
    `loom:` catalog id prefix via `cosmosIdFromLoomId`).
  - Merges measures from `item.state.content.measures` and `item.state.model.measures`.
  - Calls Azure OpenAI through `resolveBulkDescribeTarget` (tenant Copilot config
    → `LOOM_AOAI_ENDPOINT`/`LOOM_AOAI_DEPLOYMENT` env → Foundry discovery).
  - **Apply** writes descriptions onto `tables[].columns[].description` and
    `measures[].description` and `items.item(id, workspaceId).replace()`.

## Infra (already deployed — no new infra)

- Env vars `LOOM_AOAI_ENDPOINT` / `LOOM_AOAI_DEPLOYMENT` — already wired in
  `platform/fiab/bicep/main.bicep` + `modules/admin-plane/main.bicep`.
- Role: the Console UAMI's **Cognitive Services OpenAI User** on the AOAI account
  (same grant the Copilot uses) — already in the AOAI bicep module.
- Storage: existing Cosmos `items` container (`item.state.content`) — no new
  container, no new env var, no Fabric/Power BI dependency.

## Verification

`POST /api/catalog/describe` with `apply:false` returns
`{ ok:true, measures:[...], columns:[...] }`; with `apply:true` the model's
Tables/Measures tabs then show the written descriptions. With AOAI unset the
route returns `{ ok:false, aoaiUnavailable:true }` and the UI renders the honest
warning MessageBar (no fake success).
